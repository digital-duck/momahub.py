● Good question — both are relevant for the arxiv use case. Here's a practical comparison:

  pypdf

  Pros:
  - Pure Python, zero heavy dependencies — installs in seconds
  - Fast for bulk/overnight batch jobs
  - Simple API: PdfReader(bytes_io).pages[i].extract_text()
  - Good enough for single-column arXiv preprints (the LaTeX-generated ones)

  Cons:
  - Dumb text extraction — no layout understanding; two-column papers get text interleaved across columns
  - Tables come out as garbled space-separated text
  - Math/equations become noise or disappear
  - No OCR — scanned PDFs return empty strings
  - Paragraph boundaries are lost (words can run together)

  ---
  docling (IBM Research, 2024)

  Pros:
  - Layout-aware: understands columns, sections, headings — follows reading order correctly
  - Tables preserved as structured data (can export to Markdown tables or pandas)
  - Figure captions extracted and associated with figures
  - Exports to clean Markdown or JSON — much better input for an LLM
  - Built-in OCR (EasyOCR / Tesseract) for scanned PDFs
  - Handles DOCX, PPTX, HTML too

  Cons:
  - Heavy — downloads ~1–2 GB of models on first run
  - Slow: ~10–30s per paper vs <1s for pypdf
  - GPU recommended for reasonable throughput
  - Overkill for simple "dump text to LLM" workflows

  ---
  For this use case

  ┌─────────────────────────────────────────────────────┬───────────────┐
  │                      Scenario                       │    Winner     │
  ├─────────────────────────────────────────────────────┼───────────────┤
  │ Quick PoC, mostly LaTeX arXiv papers                │ pypdf         │
  ├─────────────────────────────────────────────────────┼───────────────┤
  │ Papers with tables you want the LLM to reason about │ docling       │
  ├─────────────────────────────────────────────────────┼───────────────┤
  │ Scanned / older papers                              │ docling (OCR) │
  ├─────────────────────────────────────────────────────┼───────────────┤
  │ Running overnight batch on 20+ papers               │ pypdf (speed) │
  ├─────────────────────────────────────────────────────┼───────────────┤

> thank you! can you explain how GPU nodes behind ISP firewall will communicate with momahub? I know SETI@home must solve
  this issue many years ago, but like to understand it more clearly

● Great question. This is a fundamental networking challenge for any distributed compute system. Let me explain the problem and
   solutions clearly.

  The Problem

  ┌─────────────────┐         ┌─────────────────┐
  │  GPU Node (home) │         │   Hub (cloud)    │
  │  192.168.1.10    │◄── X ──│  hub.example.com │
  │  behind NAT/FW   │         │  public IP       │
  └─────────────────┘         └─────────────────┘

  Hub CANNOT reach the GPU node — ISP firewall blocks inbound connections.
  But the GPU node CAN reach the hub — outbound connections are always allowed.

  The hub needs to send tasks to agents, but agents are behind NAT/firewalls that block unsolicited inbound traffic.

  How SETI@home Solved It: Pull Model (Polling)

  SETI@home used the simplest approach — the client always initiates:

  GPU Node                          Hub
     │                                │
     │──── "Any work for me?" ───────►│
     │◄─── "Yes, here's a task" ─────│
     │                                │
     │     ... computes for hours ... │
     │                                │
     │──── "Here are my results" ────►│
     │◄─── "Thanks, here's more" ────│

  No inbound connection needed. The client polls periodically. This works because SETI@home tasks were huge (hours of work), so
   polling every few minutes was fine.

  How momahub Currently Works: Push Model (Problem)

  Right now, momahub uses hub pushes to agents:

  # igrid/hub/dispatcher.py — hub POSTs to agent
  url = f"http://{agent['host']}:{agent['port']}/run"
  resp = await client.post(url, json=req.model_dump())

  This requires the hub to connect to the agent — works on a LAN (weekend test), but breaks across the internet because the
  agent's port 8100 is unreachable behind NAT.

  Four Solutions (Progressive)

  1. Agent-Initiated Polling (SETI@home style)

  The simplest fix. Agent periodically asks "got work?":

  Agent                              Hub
    │                                  │
    │── GET /pull_task?agent_id=X ────►│
    │◄── {task_id, model, prompt} ────│
    │                                  │
    │    ... runs Ollama locally ...   │
    │                                  │
    │── POST /submit_result {result} ─►│

  Pros: Zero firewall config, works everywhere, dead simple
  Cons: Latency = poll interval (e.g., 2-5s delay). For LLM inference tasks that take seconds anyway, this is perfectly
  acceptable.

  Interestingly, momahub-gemini already has this pattern in its spec — PulseResponse.new_task_available signals the agent
  during its regular heartbeat, then the agent calls /pull_task. The heartbeat is the poll.

  2. Long Polling / Server-Sent Events (SSE)

  Agent opens a persistent outbound connection; hub streams tasks down it:

  Agent                              Hub
    │                                  │
    │── GET /task-stream (hangs) ────►│
    │         ... connection held open ...
    │◄──── {new task payload} ────────│  (hub pushes when ready)
    │                                  │
    │── POST /submit_result ─────────►│

  Pros: Near-instant dispatch, still outbound-only
  Cons: Slightly more complex, needs reconnect logic

  3. WebSocket Reversal

  Agent opens a WebSocket to the hub. Hub sends tasks down the same socket:

  Agent                              Hub
    │                                  │
    │══ WebSocket connect ═══════════►│
    │◄═══════ bidirectional ═════════►│
    │                                  │
    │◄── "run this task" ────────────│  (hub pushes via WS)
    │── "here's the result" ────────►│  (agent responds via WS)

  Pros: True bidirectional, low latency, single connection
  Cons: More implementation work, needs WS keepalive

  This is what the momahub-sonnet SPECKIT-opus.md spec describes as "Phase 3" of WAN migration — and it's what most modern
  systems (Tailscale, Cloudflare Tunnel, etc.) do under the hood.

  4. VPN / Tunnel Overlay

  Put all nodes on a virtual private network so they appear to be on the same LAN:

  Agent (home)                    Hub (cloud)
    │                                │
    │── Tailscale/WireGuard/ZeroTier─│
    │     virtual IP: 100.64.0.2     │  virtual IP: 100.64.0.1
    │                                │
    │  Now hub can push directly —   │
    │  the current code works as-is  │

  Pros: Zero code changes to momahub, all nodes see each other
  Cons: Every participant must install the VPN client

  What I'd Recommend for momahub

  Short term (this weekend LAN test): Current push model works fine — all machines are on the same network.

  Medium term (WAN deployment): Option 1 (polling) is the lowest-effort path. The heartbeat/pulse mechanism already exists —
  extending it to carry task assignments is natural. The momahub-gemini spec (jewel #10, PulseResponse.new_task_available) was
  already heading this direction.

  Long term (production): Option 3 (WebSocket reversal) gives the best latency-to-complexity ratio. The agent opens one
  persistent WS connection to the hub, and tasks flow instantly in both directions.

  The sonnet/opus SPECKIT-opus.md already laid out this exact 4-phase roadmap (Section 17), with effort estimates: Phase 1
  (VPN) = 0 code changes, Phase 2 (HTTPS + auth) = 1-2 days, Phase 3 (WebSocket reversal) = 1 week, Phase 4 (trustless
  incentives) = 2-4 weeks.

 Implementation: SSE Pull Mode

 How it works

 PUSH mode (existing, LAN):
   Hub dispatch_loop -> pick_agent -> claim_task
     -> HTTP POST agent:8100/run -> receive TaskResult -> complete_task

 PULL mode (new, WAN-safe):
   Agent -> GET hub:8000/task-stream/{agent_id}  (SSE, stays open)
   Hub dispatch_loop -> pick_agent -> claim_task -> queue.put(task)
   SSE handler yields task JSON to agent
   Agent runs Ollama locally -> POST hub:8000/results
   Hub /results -> complete_task + record_reward
