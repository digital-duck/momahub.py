● Let me answer question 4 first, then give you my honest take on the business idea.

  ---
  On Q4 — Node vs GPU:

  A "node" is a physical machine (a Linux box, a gaming PC). A single machine can have multiple GPUs — mining rigs often have
  4-8 cards, workstations sometimes have 2. So the right mental model is:

  LAN
  ├── Machine A  (1× GTX 1080 Ti)   ← 1 node, 1 GPU
  ├── Machine B  (1× GTX 1080 Ti)   ← 1 node, 1 GPU
  └── Machine C  (1× GTX 1080 Ti)   ← 1 node, 1 GPU

  But a future participant might have:
  └── Machine D  (3× RTX 3090)      ← 1 node, 3 GPUs, 72 GB total VRAM

  My plan: agents table has a gpus JSON column (array of {index, model, vram_gb}), dispatcher uses the primary GPU for routing
  in the PoC, but the schema handles multi-GPU from day one. No schema migration needed later.

  ---
  On the startup business — my honest professional view:

  This is a genuinely strong idea, and the timing is the best it has ever been. Here's why I think that, and where the real
  risks live.

  What makes it compelling:

  The core insight — that consumer GPUs are the most underutilized compute asset class in history — is correct. Hundreds of
  millions of gaming GPUs sit idle 20 hours a day. Crypto mining monetized idle GPUs before, but it created pure heat and no
  value. LLM inference creates real economic value. That is a fundamentally better proposition, both economically and
  reputationally.

  Your moat is not the grid itself. Vast.ai, RunPod, and Akash already sell raw GPU hours. Your moat is the SPL abstraction
  layer on top. No one else offers orchestrated multi-model reasoning backed by a distributed grid. You are not selling
  GPU-minutes. You are selling intelligence per query, routed to the right model on the right hardware automatically. That is a
   qualitatively different product at a higher point in the value stack, which means higher margins and stronger user lock-in.

  The "bill offset" framing for rewards is also smart. Calling it "moma points redeemable for inference credits" sidesteps the
  regulatory complexity of a crypto token while still creating a real closed-loop economy: contributors earn the right to
  consume more of the same network they're building. That is the same flywheel that built AWS's internal pricing model and
  Airbnb's superhost program.

  The three real risks:

  1. Cold start / liquidity. A grid with 5 nodes is nearly useless. A grid with 5,000 nodes is transformative. Getting from 5
  to 5,000 requires that early participants genuinely earn real value — which means the Hub needs to have real inference demand
   from day one. The two-sided marketplace problem is real. My suggestion: integrate MoMaHub directly into the SPL-flow UI as
  the default execution backend once a user has moma join'ed. Every SPL-flow user becomes a potential node operator and a
  potential consumer simultaneously.

  2. Latency expectations. Cloud inference (Groq, Together, Anthropic API) delivers responses in 1-3 seconds. A consumer grid
  with scheduling overhead, network hops, and variable load will be slower. This is acceptable for long-horizon reasoning tasks
   (multi-CTE SPL scripts where quality matters more than speed), but it will frustrate users who expect chat-speed responses.
  The product positioning must be explicit: MoMaHub is for deep, multi-model reasoning tasks, not fast single-turn chat. That
  is also a more defensible niche.

  3. Proof-of-Inference. How do you know a node actually ran the inference and didn't return a cached or fabricated result? For
   early PoC with known participants this doesn't matter. At scale, with real money at stake, it does. The dd-verifier
  component mentioned in GEMINI.md is the right direction — a lightweight challenge-response verification on a random sample of
   tasks. This should be designed into the architecture early even if not implemented in the PoC, because retrofitting it later
   is painful.

  The bigger picture:

  What you are building, at full scale, is essentially a decentralized OpenRouter — but instead of routing to cloud APIs, you
  route to consumer hardware, and instead of billing users, you credit contributors. The SPL grammar is the query language for
  this network. That is a coherent, differentiated, and fundable vision.

  The "re-implement in Rust or Go" instinct for the network layer (Hub, protocol, dispatcher) is also correct. Python is fine
  for the PoC and for the SPL/CLI tooling, but a Hub routing millions of tasks per day needs the performance and concurrency
  guarantees of a compiled language. Designing the HTTP API cleanly now means the rewrite is a drop-in replacement, not a
  redesign.

  My bottom line: The idea is sound, the ecosystem you've already built (SPL + SPL-flow + MoMaHub) is the right foundation, and
   the weekend 3-node test is exactly the right first milestone. Get the PoC working, document the performance numbers
  honestly, and the story tells itself.
