## 1. Executive Summary

The Momahub (i-grid) codebase is a well-engineered, modern Python application that successfully abstracts distributed LLM inference into a programmable compute surface. It leverages an asynchronous "hub-and-spoke" architecture that is both flexible and resilient, particularly suited for heterogeneous LAN/WAN environments. The implementation of "Compute Tiers" and a "Hybrid Push/Pull" communication model provides a solid foundation for decentralized AI networks.

---

## 2. Architectural Strengths

- **Hybrid Communication Model:** The support for both HTTP Push (Hub-to-Agent) and SSE Pull (Agent-to-Hub) is a critical design choice. It allows agents behind restrictive NATs or firewalls to participate in the grid without complex networking setup.
- **Asynchronous Integrity:** The consistent use of `FastAPI`, `aiosqlite`, and `httpx` ensures that the Hub can handle hundreds of concurrent heartbeats and task dispatches without blocking on I/O.
- **Task State Machine:** The state management logic in `GridState` correctly handles the non-deterministic nature of distributed computing, with built-in retries and stale agent eviction (90s timeout).
- **Compute Tiering:** Normalizing performance via measured Tokens-Per-Second (TPS) allows for intelligent scheduling that respects the varying capabilities of consumer GPUs (e.g., GTX 1080 Ti vs. RTX 4090).

## 3. Identified Bottlenecks & Optimization Opportunities

### 3.1 Dispatcher Scalability (`igrid/hub/dispatcher.py`)
- **Finding:** The `pick_agent` function performs a separate database query to count active tasks for *each* candidate agent during the matching process. In a grid with $N$ agents, this results in $N$ additional DB roundtrips per task.
- **Risk:** High latency and DB contention as the agent count grows.
- **Recommendation:** Denormalize the `active_tasks` count into the `agents` table or maintain an in-memory counter in `GridState`. This reduces the agent selection complexity from $O(N_{agents} \times N_{tasks})$ to $O(N_{tasks})$.

### 3.2 Clustering Latency (`igrid/hub/cluster.py`)
- **Finding:** Task forwarding between peer hubs currently relies on an exponential backoff polling mechanism (`_poll_peer`). This can introduce a 2–10 second delay even if the remote task finishes instantly.
- **Risk:** Poor user experience for "On Grid" tasks that get forwarded.
- **Recommendation:** Implement a **Callback Webhook** or **SSE Event Notification** between hubs. When a peer hub completes a forwarded task, it should "push" the result back to the originating hub immediately.

### 3.3 SQLite Write Contention
- **Finding:** `aiosqlite` manages the single-writer constraint of SQLite well, but under high burst load (e.g., 50+ agents pulsing simultaneously), the WAL (Write-Ahead Log) may experience lock contention.
- **Risk:** Occasional `SQLITE_BUSY` errors during high-throughput stress tests.
- **Recommendation:** Introduce a configuration option to support **PostgreSQL** for large-scale deployments, while maintaining SQLite for easy "zero-config" setup.

### 3.4 Agent Resilience & Timeouts (`igrid/agent/worker.py`)
- **Finding:** The agent relies on the Ollama backend's responsiveness. If the backend stalls, the `run_task` coroutine stays open indefinitely, occupying a concurrency slot.
- **Risk:** Agent "phantom" capacity loss where it appears online but cannot process tasks.
- **Recommendation:** Add an explicit `asyncio.wait_for(..., timeout=req.timeout_s)` around the backend call to ensure the agent stays responsive and can report failures accurately.

### 3.5 Security & Reward Integrity
- **Finding:** Rewards are currently tracked via `operator_id` provided in plain text during the handshake.
- **Risk:** Potential for "spoofing" or reward redirection in public/adversarial grids.
- **Recommendation:** Implement **Cryptographic Handshakes** (e.g., Ed25519 signing) where agents prove ownership of an identity before being allowed to claim rewards.

### 3.6 Public Repository Audit (`github.com/digital-duck/momahub.py`)
- **Finding:** A manual audit of the public GitHub repository was performed to ensure no sensitive files or local state were inadvertently exposed.
- **Status:** **PASS**. The repository is clean.
- **Verified Protections:**
    - `.gitignore` successfully excludes `.igrid/`, `.spl/`, and `*.db` files.
    - No `.env` files or hardcoded credentials detected in `scripts/` or `igrid/`.
    - No private keys (`.pem`, `.key`) or sensitive certificates present.

## 4. Next Iteration Roadmap

1. **Denormalized Dispatching:** Update `GridState` to track active task counts as a property of the agent record.
2. **Reactive Forwarding:** Replace polling with a lightweight notification system for multi-hub task completion.

4. **Validation Suite:** Expand `tests/e2e/test_two_hub_grid.py` to simulate high-latency network conditions between peers.
